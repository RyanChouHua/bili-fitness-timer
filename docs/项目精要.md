# 项目精要 — B站健身计时器

> 本文从旧项目 **v0.4.17** 提炼,作为 v2 重写的唯一事实来源。旧项目已归档到
> `个人项目与资料/90_归档/B站健身计时器/`,其 `src/core.ts` 与 `timestamps/` 可直接复用。
>
> 原则:v2 只重写 UI 层,**领域逻辑与数据契约必须与本文一致**,不得随意改语义。

---

## 1. 一句话定位

Tampermonkey 用户脚本:把 B 站视频片段变成健身间歇训练。用户录入「动作 时间段 组数x次数 rest秒」,
脚本在视频上循环播放该片段,完成一组后进入休息倒计时,倒计时结束播提示音,自动进入下一组 / 下一动作。

无后端、无账号、无网络依赖(在线时间戳除外)。全部状态存 `localStorage`。

---

## 2. 核心用户流程

1. 打开含 BV 号的 B 站视频页 → 面板自动注入(非视频页不注入)。
2. 在录入框输入计划文本,多行,每行一个动作。
3. 选择「从哪个动作开始」→ 点「开始训练」。
4. 视频跳到该动作 `start`,循环播放 `[start, end]` 片段。
5. 用户做完一组 → 点「完成本组」→ 进入休息倒计时。
6. 倒计时结束 → 提示音 → 自动进入下一组(同动作)或下一动作。
7. 全部完成 → `complete` 状态。

中途可:暂停 / 继续、跳过休息、重置训练(训练中重置需二次确认)。

---

## 3. 功能清单(按优先级,v2 分阶段实现)

### MVP(第一版必须)
- 计划文本解析(见 §5)
- 面板注入,仅在含 BV 号且存在 `<video>` 的页面
- 训练状态机 + 视频片段循环(§8)
- 休息倒计时 + Web Audio 提示音
- 「从指定动作开始」
- 计划按 BV 号持久化到 `localStorage`
- 完成本组 / 暂停 / 继续 / 跳过休息 / 重置(训练中重置二次确认)

### P1(重要)
- 从当前视频时间「插入开始 / 插入结束」
- 子分组管理:一个 BV 下多个子分组(周一 / 周二…),新建 / 复制 / 切换 / 改名 / 删除
- 本地 JSON 导入 / 导出
- 按 BV 在线导入 GitHub raw 时间戳
- 动作预览列表 + 锁定 / 解锁(锁定时训练中不可切换动作)
- 计划标题 / 作者 / 备注

### P2(体验,可后置)
- 面板收起 / 展开
- 面板拖动、缩放(**v2 重新评估,见 §10 教训**)
- 分组分页(旧版 `groupPageSize = 4`)

### 竞品对照后新增候选(2026-07-23 调研)

对照两个同类脚本(GreasyFork 的 [Bilibili Loop Clip](https://greasyfork.org/zh-TW/scripts/549514-bilibili-loop-clip) 片段循环、以及一个 Grok 生成的「B 站健身计时循环助手」HIIT 秒表)后提炼。这些是**候选**,不改现有界面架构,只往训练舱/工作台里加内容,不阻塞卡 2R 预览门。

- **视频叠加大倒计时(建议提到 MVP/P1)**:休息倒计时以视频上叠加的**超大数字**呈现,而非只在面板里显示小字。直接服务「余光可读」的核心设计目标;训练舱已在视频层,实现成本低。见 [界面架构.md](界面架构.md) §2 训练舱。
- **一键预设 / 示例模板(P1)**:提供 2–3 个一键填入的示例(如旧版「示例」按钮、或 HIIT/Tabata 风格),降低文本录入门槛。放工作台。**不做**复杂预设管理。
- **休息时自动暂停视频(P2,设置项)**:休息阶段自动暂停视频、下一组开始自动续播,省得视频空转。做成可开关的设置。
- **训练全局进度(P1)**:训练舱现在只显示「第 2/3 组」「动作 2/4」这类局部进度,看不到整个 session 还剩多少。加一条极细的顶部总进度(已完成组数 / 总组数,可选预估剩余时间)。长计划跟练时给用户「盼头」,直接服务跟练体验。**零新增数据**——总组数由 exercises 求和、已完成数由 runtime 推导;一个组件即可,不碰状态机。放训练舱。
- **断点续练(P2,持久化)**:当前 runtime 是纯内存态(见 §8),刷新 / 误触 / 切视频查动作后训练进度全丢,得从头开始。将当前 runtime 也落盘 localStorage(与计划库分开的 key,如 `bili-fitness-timer:session`),重进 B 站视频页且 BV 匹配时提示「上次训练到『俯卧撑 第 2/3 组』,继续吗?」,可继续或放弃。补的是「跟到一半丢进度」这个真实痛点。**注意**:此项要求 Wave 2 持久化设计**预留 session 存储结构**,否则事后回改 store 成本高;runtime 序列化需与 §8 状态机字段一致,恢复时校验 exercises 未变(计划被改过则不恢复)。**不做**跨设备同步、不做历史记录(见下「明确不做」)。

**守住的差异化(不为简化而砍)**:文本录入 + 在线时间戳库(GitHub)、多子分组、「完成本组」手动推进。这三点是同类脚本都没有的护城河——竞品要么是纯 A-B 循环,要么是纯 HIIT 秒表,都没有「动作序列 + 组次 + 计划库」这套「把健身视频变成结构化力量训练课表」的定位。

**明确不做**(避免过度设计):A-B 时间轴可视化拖拽选段(已有「插入当前时间」够用)、复杂预设管理系统。

## 4. 数据模型(与旧 `core.ts` 一致,不改)

```ts
interface Exercise {
  id: string
  name: string
  start: number        // 秒
  end: number          // 秒,必须 > start
  sets: number         // > 0
  minReps: number      // > 0
  maxReps: number      // >= minReps
  restSeconds: number  // >= 0
}

interface Settings {
  beepDuration: number       // 提示音时长,可选 1 / 2 / 3 / 5,默认 2
  pauseDuringRest: boolean    // 默认 true
}

interface StoredPlan {           // 一个子分组
  id: string
  rawInput: string
  settings: Settings
  savedExercises: Exercise[]
  bvid?: string | null
  title?: string
  author?: string
  notes?: string
  createdAt?: number
  updatedAt?: number
}

interface StoredPlanLibrary {     // 一个 BV 视频分组
  schemaVersion: 2
  bvid?: string | null
  activeGroupId: string
  groups: StoredPlan[]
  updatedAt?: number
}

interface Preferences {           // 全局偏好
  panelPosition: { left: number; top: number } | null
  panelSize: { width: number; height: number } | null
  previewLocked: boolean
  activeTab: 'groups' | 'preview' | 'settings'
  inputCollapsed: boolean
}
```

运行时(不持久化):

```ts
type Mode = 'idle' | 'exercise' | 'rest' | 'paused' | 'complete'
interface Runtime {
  mode: Mode
  exerciseIndex: number
  setIndex: number
  restRemaining: number
  beforePauseMode: Exclude<Mode, 'paused'>
}
```

## 5. 计划文本解析规则(`parsePlan`,逐行)

每行独立解析,空行跳过。一行需同时满足「时间段」和「组数次数」,否则报错并跳过该行。

- **时间段**(必需):`<start>` `(-|~|至|到)` `<end>`,如 `00:12-00:28`、`1:05~1:22`。`end` 必须 > `start`。
- **组数次数**(必需):`<sets>` `(x|X|×|组)` `<min>` `([-~]<max>)?`,如 `3x8-12`、`4×10`、`3组8`。默认 `max = min`。
- **休息**(可选):`(rest|休息)<秒>`,如 `rest45`、`休息60`。缺省 **45 秒**。
- **动作名**:时间段之前的文本;为空则 `动作 N`。

### 时间戳格式(`parseTimestamp`)
- `m:ss` / `h:mm:ss` / `mm:ss.d`(小数秒)
- 中文:`1时`、`5分`、`30秒`、`1时5分30秒`、`5分30`、`90秒` 等组合。

### 序列化(`serializeExercises`)
`{name} {mm:ss}-{mm:ss} {sets}x{min}[-{max}] rest{restSeconds}`,`max===min` 时省略 `-{max}`。

## 6. 存储契约

- 前缀:`bili-fitness-timer:`
- 计划 key:`bili-fitness-timer:<BVID>`,无 BV 时用 `location.pathname` 兜底。
- 偏好 key:`bili-fitness-timer:preferences`
- 判定计划 key:以前缀开头且 ≠ 偏好 key。
- 存储值:`StoredPlanLibrary`(`schemaVersion: 2`)。
- **旧数据迁移**:旧版单计划对象(无 `groups`)通过 `createLibraryFromLegacy` 包装成单子分组。导入 / 摘要函数(`normalizeImportedPlanData`、`summarizeStoredPlan`)都保留 legacy 兼容分支,v2 不能丢。

## 7. 在线时间戳格式

请求地址:
```
https://github.com/RyanChouHua/bili-fitness-timer/raw/refs/heads/main/timestamps/<BV号>.json
```

格式:
```json
{
  "bvid": "BV1TT4y1f7A3",
  "title": "视频分组标题",
  "author": "...",
  "notes": "...",
  "groups": [
    { "title": "周一动作", "rawInput": "俯卧撑 00:12-00:28 3x8-12 rest45" },
    { "title": "周二动作", "rawInput": "深蹲 01:05-01:22 4x10 rest60" }
  ]
}
```

- 每个 `groups[]` = 一个子分组。优先 `rawInput`,没有则从 `exercises` 生成。
- 旧版单计划 JSON(无 `groups`)兼容为单子分组。
- **404 / 网络失败 / JSON 错误时不覆盖本地数据**。

## 8. 运行时状态机

模式:`idle | exercise | rest | paused | complete`

转移:
- `idle` --开始训练--> `exercise`(视频跳到 `start`,循环播放片段)
- `exercise` --完成本组-->
  - 还有组 / 还有动作 → `rest`(启动倒计时)
  - 全部完成 → `complete`
- `rest` --倒计时归零--> 播提示音 → `moveToNextUnit` → `exercise` 或 `complete`
- `rest` --跳过休息--> `moveToNextUnit`
- `exercise`/`rest` --暂停--> `paused`(记录 `beforePauseMode`)
- `paused` --继续--> 恢复 `beforePauseMode`
- 任意 --重置--> `idle`(训练中需二次确认)

`moveToNextUnit`:先递增 `setIndex`,超过该动作 `sets` 则动作 `exerciseIndex + 1`、`setIndex` 归零;越过最后动作 → `complete`。

## 9. 关键技术约束(平台集成,v2 必须原样保留行为)

- `@grant none`、`@run-at document-idle`。
- `@match`:`https://www.bilibili.com/*`、`https://m.bilibili.com/*`、`https://bilibili.com/*`。
- **SPA 导航监听**:`setInterval` 每 1s 检查当前 BV。BV 消失 → 拆面板;BV 变化 → 换存储 key、重载计划、重绑视频。
- **视频片段循环**:对 `<video>` 绑 `timeupdate`,`exercise` 模式下超过 `end` 或早于 `start-0.25` 则跳回 `start` 并续播。用 `WeakSet` 防重复绑定。
- **等待视频**:`waitForVideo` 用 `MutationObserver`,超时 10s。
- **提示音**:Web Audio,880Hz 正弦,`beepDuration` 秒淡出。
- 面板 `z-index: 2147483647`,`position: fixed`。

## 10. 分发模型

- 构建产物 `dist/bili-fitness-timer.user.js`(带 UserScript 头)+ `dist/bili-fitness-timer.meta.js`。
- 两者都提交到 `main`,Tampermonkey 通过 `@updateURL`(meta)+ `@downloadURL`(user.js)自动更新。
- 版本号三处一致:`package.json`、`package-lock.json`、`vite.config.ts` 的 `@version`。
- 安装入口:GitHub raw `dist/bili-fitness-timer.user.js`。

> v2 若换新仓库名 / 新安装地址,需在重构计划中明确;若沿用旧仓库,注意 `@updateURL` 会让旧用户平滑升级到 v2。

### v2 发布决策(2026-07-23 拍板,沿用旧仓库平滑升级)

沿用旧项目仓库 `github.com/RyanChouHua/bili-fitness-timer`,不新建。三项已定:

## 11. 旧项目痛点与教训(v2 必须规避)

这些是「面板一直弄不好」的根因,逐条对应 v2 设计原则。

| 痛点 | 根因 | v2 对策 |
|---|---|---|
| textarea 输入时焦点 / 光标丢失 | `render()` 每次 `replaceChildren` 全量重建 DOM,靠 `restoreTextarea` 手动补救 | 声明式渲染,由 reconciler(Preact)保持 DOM 与焦点 |
| CSS 无法维护 | `injectStyle()` 是 730 行手写 CSS 模板字符串 | CSS 独立文件,`?inline` 注入;设计令牌化 |
| 面板尺寸 / 位置在不同设备反复出问题 | 三套响应式(`@media 640` / `@media 641-1180` / `@container`)+ `dvh` + `env(safe-area)` + 拖动 + 缩放 + `panelSize` 持久化互相打架 | **统一单列布局 + 极少断点**,砍掉 container query 与手动 clamp 尺寸 |
| 改一处牵动全身 | `main.ts` 2701 行,UI / 状态 / DOM / 事件 / 平台集成全耦合 | 分层:`core`(纯逻辑)/ `state`(store)/ `ui`(组件)/ `platform`(B站适配),互不越界 |
| 全量重渲染性能 & 交互抖动 | 命令式 300 行 `render()` 每次重建 | 组件级 diff,只更新变化部分 |

**保留的好东西**:`core.ts` 已是纯函数、无 DOM 依赖、可被 `node --check` / 断言直接测。v2 原样复用,只补类型与测试。
