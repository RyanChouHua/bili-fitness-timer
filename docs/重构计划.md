# 重构计划 — bili-fitness-timer-v2

> 配套文档:[项目精要.md](项目精要.md)(事实来源)、[codex任务分配.md](codex任务分配.md)(给执行者的任务卡)。
>
> 角色分工:**Claude 负责规划与任务分配 + 阶段验收**,**Codex 负责具体脚本编写**。
>
> 工作流:vibe coding —— 先立骨架 → **界面 HTML 预览先行,通过后再接功能** → 分层实现 → 集成验证。

---

## 1. 目标与非目标

### 目标
- 保留旧项目全部数据契约与领域逻辑(见精要 §4–§9),只重写 UI 层与工程结构。
- 用**声明式渲染**解决焦点丢失 / 全量重建问题。
- 用**统一单列布局 + 极少断点**解决多设备面板反复出问题。
- 代码分层清晰,单个文件不再膨胀到 2700 行。

### 非目标
- 不改计划文本语法、存储 key、在线时间戳格式(否则老用户数据 / 时间戳库失效)。
- 不加后端、账号、云同步。
- MVP 不追求功能超过旧版;先稳,再补 P1 / P2。

---

## 2. 技术选型(已定)

| 维度 | 选择 | 理由 |
|---|---|---|
| 语言 / 构建 | TypeScript + Vite(沿用) | 与旧项目一致,构建 / meta 插件可复用 |
| UI 渲染 | **声明式**(Preact + htm,`@grant none` 友好,压缩后体积小) | 解决焦点丢失与全量重建;无需 JSX 编译额外配置(htm 用模板字符串) |
| 状态管理 | 轻量集中式 store(自写 `createStore` 或 Preact signals) | 单一数据源,UI 订阅;避免散落全局变量 |
| 样式 | 独立 `.css` 文件,Vite `?inline` 注入 `<style>` | 摆脱 730 行模板字符串;设计令牌化 |
| 布局 | 统一单列 + 1–2 个断点 | 见 §5 |

> Preact + htm 走 CDN import 还是打包进 bundle:**打包进 bundle**(用户脚本要单文件、离线可用、`@grant none` 下不便动态加载)。Vite lib 模式 `inlineDynamicImports` 已支持。

│   │   ├── parse.ts            # parsePlan / parseTimestamp / formatTimestamp / serialize

│   ├── platform/
│   │   ├── bilibili.ts         # BV 提取、waitForVideo、导航监听、片段循环守卫

├── timestamps/                 # 从旧项目复制,保持格式
├── scripts/
│   ├── check.mjs               # core 单测(迁移 + 扩充)
│   └── screenshot-tablet.mjs   # Playwright 截图(迁移)

**分层边界(硬约束)**:
- `core/` 不 import `state`/`ui`/`platform`,不碰 `window`/`document`/`localStorage`。
- `platform/` 只做 B站 DOM 与浏览器 API 适配,不含业务判断。
- `ui/` 只读 store、派发 action,不直接操作视频或 localStorage。
- `state/` 是唯一把 core + platform + storage 粘起来的地方。

## 4. 分阶段执行(vibe coding)

> 每个阶段有明确产出与验收门。**阶段 2 的 HTML 预览必须经用户确认通过,才进入阶段 3 功能实现**(用户明确要求)。

### 阶段 0 — 脚手架(Codex)
- 建目录结构、`package.json`(含 Preact/htm 依赖)、`vite.config.ts`(迁移 userscript meta 插件 + inline CSS)、`tsconfig.json`。
- 从旧项目复制 `timestamps/`。
- **验收**:`npm install` 成功,`npm run build` 产出空壳 user.js + meta.js,版本号三处一致。

### 阶段 1 — core 迁移(Codex)
- 把旧 `src/core.ts` 拆进 `src/core/`(model/parse/storage/plan),补状态机纯函数 `moveToNextUnit`。
- 迁移并扩充 `scripts/check.mjs`,覆盖精要 §5–§8 的规则。
- **验收**:`npm run test` 全绿,`npm run typecheck` 无错。core 零 DOM 依赖。

### 阶段 2 — 界面 HTML 预览(Codex → 用户验收 ★门)
- 只做 UI 组件 + 样式 + `preview/index.html`,喂 `mock-data.ts` 假数据。
- 覆盖全部状态:`idle / exercise / rest / paused / complete`,含错误录入、空动作、多子分组、移动 / 平板 / 桌面三种宽度。
- `preview/index.html` 提供状态切换器,便于用户点选查看。
- **验收(用户)**:`npm run preview`(vite dev server)打开预览,用户逐状态确认布局与视觉。
  - 通过 → 进入阶段 3。
  - 不通过 → Claude 收集反馈,更新任务卡,Codex 迭代,重新验收。
- **此门未过,不写任何功能逻辑。**

### 阶段 3 — 状态与功能接线(Codex)
- 实现 `state/store.ts`:actions 调用 core 纯函数 + platform 副作用。
- 组件从假数据切换到订阅 store。
- MVP 功能全通(精要 §3 MVP)。
- **验收**:`typecheck` + `test` 通过;手动/Playwright 在真实 B站页跑通完整训练流程。

### 阶段 4 — 平台集成与持久化(Codex)
- `platform/bilibili.ts`:BV 提取、waitForVideo、SPA 导航监听、片段循环守卫。
- `state` 接 `localStorage`,含 legacy 迁移。
- **验收**:切视频 / 刷新 / 收起展开后状态正确;旧版 localStorage 数据能被读起来。

### 阶段 5 — P1 功能(Codex)
- 子分组管理、导入导出、在线导入、预览锁定、计划信息表单、插入时间。
- **验收**:逐项对照精要 §3 P1。

### 阶段 6 — 发布准备(Claude 把关 + Codex)
- README、版本记录、截图、meta/user.js 提交策略。
- 决定仓库 / 安装地址(见精要 §10 待决)。
- **验收**:构建产物完整,文档齐全。

## 5. 布局与样式设计原则(解决面板痛点)

> ⚠️ 本节在完成卡 2 首版预览后,依据用户决策**重定义为「三区域接管式」架构**。
> 完整设计见 **[界面架构.md](界面架构.md)**(界面层唯一事实来源)。以下为要点,与界面架构文档冲突时以后者为准。

**核心思路**:不再是「一个悬浮窗塞所有功能」,而是**按使用阶段拆分**——准备计划与进行训练是两种相反的使用场景(信息密集 vs 极简),硬塞一个面板正是旧版反复出问题的根源。拆成三个区域:

1. **训练舱(训练阶段)**:覆盖层接管 B 站视频页,**只保留视频框 + 评论区**,其余原生区域全遮罩。只放训练必需:当前动作、组次进度、超大主按钮(完成本组/继续/休息倒计时)、次要控制(暂停/跳过)、动作队列迷你预览、安全区(重置需二次确认)。**无录入、无设置**。
2. **计划工作台(准备阶段)**:从侧边滑出的**抽屉**,不与视频抢空间。含录入区 + 解析错误、插入当前时间、分组管理(列表/新建/复制/切换/改名/删除/分页)、计划信息(标题/作者/备注)、导入导出/在线导入、设置(提示音/锁定预览)。
3. **常驻控制条**:极小贴边条,负责切换两个形态 + 显示当前分组名。

**关键约束**:
- **训练舱与工作台互斥**,一次只用一个。
- **全去掉**旧版的拖动、手动缩放、收起展开(这些是旧版最易坏处),靠固定布局。
- **移动端/平板**:训练舱贴底部、视频在上;工作台抽屉全屏或近全屏。这是旧版最易破版处,新架构用「视频在上、控制贴底」的固定纵向堆叠规避。
- **极少断点**:砍掉旧版 `@container` query、`641-1180` 平板专用段、手动 `clamp` 尺寸;最多桌面/移动一个断点。
- **设计令牌**:颜色/圆角/间距/字号用 CSS 变量集中定义。
- **触控目标**:按钮最小高度 ≥ 44px(训练舱主按钮更大),主操作居中大按钮。

## 6. 风险与应对

| 风险 | 应对 |
|---|---|
| Preact + htm 打包后与 `@grant none` / B站 CSP 冲突 | 阶段 0 用最小 demo 在真实 B站页验证注入可行,再铺开 |
| B站 SPA 导航 / video 元素结构变化 | platform 层集中封装选择器与监听,变化只改一处 |
| 老用户数据迁移丢失 | 阶段 4 用旧版真实 localStorage 样本回归测试 legacy 分支 |
| 预览门反复不过拖慢进度 | Claude 在任务卡里给出明确验收清单(§4 阶段2),减少来回 |
| user.js 体积因 Preact 变大 | 可接受(仍是单文件几十 KB);必要时评估 preact/compat 裁剪 |

## 7. 验证命令(贯穿全程)

```bash
npm install
npm run test        # core 单测(node --check + 断言)
npm run typecheck   # tsc --noEmit
npm run build       # 产出 dist/*.user.js + *.meta.js
npm run preview     # 阶段2:vite dev,打开 preview/index.html
npm run screenshot:tablet   # Playwright 截图(阶段6)
```

任何阶段交付前,Codex 必须跑 `test` + `typecheck`,涉及 UI 的跑 `build`。

## 8. Claude / Codex 协作循环

1. Claude 按阶段拆任务卡(见 codex任务分配.md),一次派发一个阶段。
